Ask your codebase anything. Get evidence, not guesses.

A read-only AI agent that inspects any Git repository using three typed tools and an observe → act → reflect loop. Every answer comes with file-and-line citations. No hallucination. No write access. No network.

How it works

Observe, act, reflect, answer.

The agent doesn't follow a hard-coded sequence. It decides which tool to call, receives the result, and reflects on whether it has enough evidence — or needs to dig deeper.

1
Observe Restate the question as concrete evidence to find
→
2
Act List, read, or search — cheapest useful call first
→
3
Reflect Enough evidence? If not, act again with new scope
→
4
Answer Lead with conclusion, cite every claim with paths

Features

Safety through architecture, not prompts.

The important controls live outside the model: narrow capabilities, path confinement, bounded output, and a finite inspection budget.

🔒

Read-only tools

Three typed tools — list_files, read_file, search_code — use Node's read-only filesystem APIs. No shell, no Git, no network.

📐

Path confinement

Three-layer defense: reject .. and absolute paths at input, canonical-resolve, then verify against repo root. The model never sees the absolute path.

⏱️

Inspection budget

Shared 1–20 call budget across all three tools. After exhaustion, the agent must answer from collected evidence — no infinite loops.

📄

Evidence citations

Every claim must cite repository-relative file paths and line ranges. The agent explicitly cannot invent file contents or architecture.

🏖️

Restricted sandbox

Flue's default filesystem and shell tools are removed. The agent can only access the repository through the three custom tools.

🧪

Negative-test aware

Asking "Where is authentication?" against a repo without it produces "I searched X and found nothing" — not a hallucinated OAuth system.

Architecture

Four layers, clean separation.

Agent, tools, repository reader, and sandbox are distinct, independently testable modules.

Security

Repository access is application-controlled.

The model requests file operations by name. The application validates, confines, and caps every access before the filesystem is touched.

1

Only repository-relative paths accepted

Absolute paths and .. traversal are rejected at input validation, before any filesystem call.

2

Symlink resolution checked

Canonical path verified against repo root after realpath. Symlinks pointing outside the repo are blocked.

3

Output caps enforced

400 lines per read, 50 matches per search, 1 MB file limit. Directory walks skip symlinks, VCS, and build artifacts.

4

Content treated as untrusted data

Repository file content is never interpreted as instructions. The model is explicitly told to ignore instructions found in files.

Get started

Two clones, one question.

Point the agent at any local Git repository. The default target is project-oak/oak, a Rust monorepo for privacy-preserving distributed systems.

1

Clone both repositories as siblings

git clone https://github.com/jellydn/flue-repo-assistant.git
git clone --depth 1 https://github.com/jellydn/oak.git
2

Install dependencies and set your API key

cd flue-repo-assistant
npm install
cp .env.example .env
# add your OPENROUTER_API_KEY to .env
3

Ask a question

npm start -- --input '{"message":"What is the architecture of oak?"}'

Requires Node.js ≥ 22.19.0. Override REPOSITORY_PATH to inspect any local checkout.
